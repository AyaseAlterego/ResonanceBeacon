## 2. 六智能体角色

### 2.1 CTO 智能体（编排者）

| 属性 | 值 |
|------|-----|
| 职责 | 编排所有智能体，监控看板，选择任务，汇报状态 |
| Kanban 列 | 所有列（全局视野） |
| 核心方法 | `选择任务()` — 从 Backlog 选择最高优先级任务 |
| 核心方法 | `生成状态报告()` — 每日 CTO 日报 |
| 核心方法 | `协调开发流程()` — 编排 Dev→Security→QA→审批 |

### 2.2 PM 智能体（分诊员）

| 属性 | 值 |
|------|-----|
| 职责 | 分诊 GitHub Issues 和用户请求，评分优先级，管理 Backlog |
| Kanban 列 | backlog |
| 核心方法 | `分诊任务()` — 分析任务类型、复杂度、优先级 |
| 核心方法 | `批量分诊()` — 批量处理多个任务 |

### 2.3 Dev 智能体（开发者）

| 属性 | 值 |
|------|-----|
| 职责 | 执行开发任务，选择执行引擎，创建 PR |
| Kanban 列 | in_progress |
| 核心方法 | `执行开发()` — 选择引擎 + 构建上下文 + 调度执行 |
| 核心方法 | `创建PR()` — 创建 Pull Request |
| 引擎选择 | 单文件修复→Codex，多文件开发→OpenCode/Claude Code |

### 2.4 Security 智能体（安全官）

| 属性 | 值 |
|------|-----|
| 职责 | 密钥扫描、OWASP 检查、CVE 审计、供应链安全 |
| Kanban 列 | review（在 Dev 和 QA 之间） |
| 核心方法 | `安全审查()` — 扫描密钥 + OWASP + CVE |
| 检查项 | AWS Key、API Key、Private Key、SQL 注入、XSS、命令注入 |

### 2.5 QA 智能体（质检员）

| 属性 | 值 |
|------|-----|
| 职责 | 构建检查、健康检查、代码质量审查、生成变更摘要 |
| Kanban 列 | review |
| 核心方法 | `质量验证()` — 构建检查 + 健康检查 + 变更摘要 |
| 核心方法 | `生成审批消息()` — 生成 YES/NO 审批消息 |

### 2.6 Ops 智能体（运维员）

| 属性 | 值 |
|------|-----|
| 职责 | 部署应用、监控生产、处理事故 |
| Kanban 列 | done |
| 核心方法 | `部署()` — 部署应用（预留接口） |
| 核心方法 | `健康监控()` — 监控部署后健康状态 |

## 3. 自主循环引擎

### 3.1 循环流程

```
定时触发 (每小时)
       │
       ▼
┌──────────────┐
│ 扫描外部事件  │  ← GitHub Issues / 用户消息 / 定时任务
──────┬───────┘
       │
       ▼
┌──────────────┐
│ PM Agent     │  ← 评分分诊，创建 Kanban 卡片
│ 分诊新任务    │
──────┬───────┘
       │
       ▼
┌──────────────┐
│ CTO Agent    │  ← 从 Backlog 选择最高优先级卡片
│ 选择任务      │
└─────────────┘
       │
       ▼
┌──────────────┐
│ Dev Agent    │  ← 调用执行层，调度智能体开发
│ 执行开发      │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ Security     │  ← 密钥扫描 + OWASP + CVE
│ 安全审查      │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ QA Agent     │  ← 构建检查 + 健康检查 + 摘要
│ 质量验证      │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│ 推送审批      │  ← Web 通知 + 等待用户 YES/NO
└──────┬───────┘
       │
   YES │ NO
   ┌──────┐
   ▼       ▼
┌───── ┌────────┐
│合并  │ │ Dev    │
│部署  │ │ 迭代   │
└─────┘ └────────┘
```

### 3.2 循环状态

| 状态 | 描述 |
|------|------|
| `idle` | 空闲，等待下一次扫描 |
| `scanning` | 扫描外部事件 |
| `triaging` | PM 分诊中 |
| `selecting` | CTO 选择任务中 |
| `developing` | Dev 执行开发中 |
| `reviewing_security` | Security 安全审查中 |
| `reviewing_qa` | QA 质量验证中 |
| `awaiting_approval` | 等待用户审批 |
| `deploying` | Ops 部署中 |
| `paused` | 已暂停 |
| `error` | 错误状态 |

### 3.3 配置项

```python
@dataclass
class 自主循环配置:
    扫描间隔秒: int = 3600          # 默认每小时扫描一次
    最大并发任务: int = 3           # 最大并发任务数
    自动审批: bool = False          # 是否自动审批（开发阶段建议关闭）
    启用GitHub集成: bool = False    # 是否监听 GitHub Issues
    GitHub仓库: str = ""            # GitHub 仓库地址
    GitHub令牌: str = ""            # GitHub 访问令牌
    通知渠道: str = "web"           # 通知渠道：web / telegram / slack
```

## 4. Kanban 状态机

### 4.1 状态定义

| 状态 | 英文 | 描述 |
|------|------|------|
| 待办 | `backlog` | 任务已创建，等待被选择 |
| 进行中 | `in_progress` | Dev 正在执行开发 |
| 审查中 | `review` | 安全审查 + 质量验证，等待审批 |
| 已完成 | `done` | 已合并部署 |
| 已取消 | `cancelled` | 任务取消 |

### 4.2 状态转换规则

```
backlog ──→ in_progress ──→ review ──→ done
   │            │            │
   │            └──→ backlog └──→ in_progress
   │
   └──→ cancelled
```

### 4.3 卡片数据模型

```python
@dataclass
class Kanban卡片:
    ID: str                          # "card-{uuid4}"
    项目ID: str                      # FK → 项目
    标题: str
    描述: str = ""
    状态: str = "backlog"            # backlog/in_progress/review/done/cancelled
    优先级: str = "medium"           # critical/high/medium/low
    负责人: str = "dev"              # cto/pm/dev/security/qa/ops
    来源: str = "manual"             # manual/github_issue/scheduled
    复杂度: str = "medium"           # low/medium/high
    预估工时: str = "未知"
    等待审批: bool = False
    关联任务ID: str = ""
    关联制品ID列表: list[str] = field(default_factory=list)
    创建时间: str = ""
    更新时间: str = ""
    完成时间: str = ""
    元数据: dict = field(default_factory=dict)
```

## 6. API 设计

### 6.1 看板路由 — `/看板/`

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/看板/项目/{项目ID}` | 获取项目完整看板状态 |
| POST | `/看板/卡片` | 创建新卡片 |
| GET | `/看板/卡片/{卡片ID}` | 获取卡片详情 |
| PUT | `/看板/卡片/{卡片ID}` | 更新卡片 |
| POST | `/看板/卡片/{卡片ID}/状态` | 转换卡片状态 |
| DELETE | `/看板/卡片/{卡片ID}` | 删除卡片 |
| GET | `/看板/卡片/{卡片ID}/历史` | 获取状态转换历史 |

### 6.2 审批路由 — `/审批/`

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/审批/待处理` | 获取待处理审批（传统） |
| GET | `/审批/自主循环/待审批` | 获取自主循环待审批卡片 |
| POST | `/审批/自主循环/{卡片ID}/审批` | 提交 YES/NO 审批响应 |
| GET | `/审批/历史` | 获取审批历史 |

### 6.3 项目路由 — `/项目/`

| 方法 | 路径 | 描述 |
|------|------|------|
| GET | `/项目/` | 列出所有项目 |

| GET | `/项目/{id}` | 获取项目详情 |
| PUT | `/项目/{id}/阶段` | 更新项目阶段 |

## 12. 风险与缓解

| 风险 | 缓解措施 |
|------|---------|
| 自主循环误操作 | 默认关闭自动审批，所有变更需人工 YES/NO 确认 |
| 智能体执行失败 | 熔断器 + 重试 + 通知用户手动干预 |
| Kanban 状态不一致 | 状态机严格守卫，记录所有转换历史 |
| 安全审查漏报 | 多层检查：密钥扫描 + OWASP + CVE，不通过则阻塞 |
| 对话历史 token 超限 | 滑动窗口：最近 N 条消息完整 + 更早消息摘要 |
| GitHub 集成复杂 | Phase 1 先用内存存储和手动创建，后续再接入 GitHub API |
