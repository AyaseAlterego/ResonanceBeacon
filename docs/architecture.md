## 1. 系统概述

起源信标（ResonanceBeacon）是一个智能流水线开发系统，核心思想是让 Hermes Agent 作为元智能体，调度 Claude Code、OpenCode、Codex 等 AI 工具以流水线方式完成软件开发任务。

系统围绕"类别路由"这一核心概念设计：用户不需要指定使用哪个 AI 工具，而是通过描述任务的类别（如"深度思考"、"快速修复"、"架构设计"），由系统自动选择最合适的智能体执行。这一设计将任务需求与具体实现解耦，使系统具备良好的可扩展性。

### 核心能力

- **流水线编排**：支持顺序/并发阶段、DAG 依赖调度、状态机驱动的执行
- **智能体路由**：基于类别的自动智能体选择，支持 8 个类别
- **五层钩子**：会话、工具守护、转换、继续、技能五层生命周期钩子
- **容错保障**：安全创建、熔断器、并发管理、重试、循环检测
- **人机协作**：审批系统、访谈模式、超时自动处理
- **可观测性**：Prometheus 指标、健康检查、组件状态跟踪

## 3. 模块详解

### 3.1 编排器

**路径**: `src/hermes/编排器/`

**职责**: 流水线的执行编排，管理阶段和任务的调度顺序、状态转换和生命周期事件。

**核心组件**:

| 组件 | 文件 | 职责 |
|------|------|------|
| `流水线引擎` | `引擎.py` | 编排流水线执行，协调阶段和任务调度 |
| `流水线状态机` | `状态机.py` | 管理流水线/阶段/任务的状态转换 |
| `DAG调度器` | `调度器.py` | DAG 依赖解析、拓扑排序、并发分组 |

**流水线引擎** (`引擎.py:30`) 是编排器的入口，接收流水线定义和用户输入，按顺序执行各阶段。每个阶段支持 `sequential`（顺序）和 `parallel`（并发）两种类型。并发阶段使用 `asyncio.gather` 并行执行任务。引擎在执行过程中通过钩子组合器触发生命周期事件。

**状态机** (`状态机.py:7`) 定义了 6 种状态和它们之间的合法转换：

```
pending ──→ running ──→ waiting_approval ──→ running
              │                                    │
              ├──→ completed (终态)                 │
              ├──→ failed ──→ running (允许重试)     │
              └──→ cancelled (终态)                  │
```

**DAG调度器** (`调度器.py:12`) 基于 Kahn 算法实现拓扑排序，提供：
- `从定义构建()`: 从流水线定义构建 DAG
- `检测循环()`: 检测依赖循环
- `拓扑排序()`: 返回合法执行顺序
- `获取并发组()`: 返回可并行执行的分组

**接口依赖**: 编排器 → 智能体注册表、类别路由器、后台管理器、钩子组合器

### 3.2 智能体

**路径**: `src/hermes/智能体/`

**职责**: 管理所有 AI 工具的接入，提供统一适配器接口、基于类别的路由选择、健康检查和负载均衡。

**核心组件**:

| 组件 | 文件 | 职责 |
|------|------|------|
| `智能体适配器` (ABC) | `基础.py:72` | 所有智能体的抽象基类 |
| `智能体工厂` (ABC) | `基础.py:143` | 智能体工厂抽象基类 |
| `智能体注册表` | `注册表.py:9` | 智能体注册、发现、查询 |
| `类别路由器` | `类别.py:9` | 基于类别的智能体选择 |
| `健康检查器` | `健康检查.py:49` | 定期健康检查、负载统计 |
| `上下文管理器` | `上下文管理器.py:48` | 跨阶段上下文传递、令牌预算 |
| `提示词模板管理器` | `提示词模板.py:18` | 动态提示词生成 |

**智能体适配器接口** (`基础.py:72`) 定义了所有智能体必须实现的方法：

```python
class 智能体适配器(ABC):
    智能体ID: str              # 唯一标识
    能力列表: list[智能体能力]  # 能力描述
    类别: 智能体类别            # 主要类别
    成本层级: 智能体成本        # 成本层级

    async def 初始化(配置)          # 初始化
    async def 健康检查() -> bool    # 健康检查
    async def 执行任务(...) -> 任务结果   # 执行任务
    async def 流式执行任务(...) -> AsyncIterator  # 流式执行
    async def 取消任务(任务ID) -> bool       # 取消任务
```

**8 个智能体类别** (`基础.py:10`)：

| 类别 | 值 | 适用场景 | 默认路由 |
|------|------|----------|----------|
| 超级大脑 | `ultrabrain` | 架构级，高难度逻辑 | claude_code |
| 深度 | `deep` | 深度思考的复杂任务 | claude_code, opencode |
| 快速 | `quick` | 小改动，快速修复 | claude_code, opencode |
| 视觉工程 | `visual-engineering` | 前端/UI/UX | claude_code |
| 专家 | `specialist` | 特定领域专长 | claude_code, codex |
| 探索 | `exploration` | 代码库分析，需求理解 | claude_code, opencode |
| 顾问 | `advisor` | 审查、架构建议 | claude_code |
| 工具 | `utility` | 测试、文档、配置 | claude_code, opencode, codex |

**类别路由选择策略** (`类别.py:41`)：
1. 从路由表获取候选智能体列表
2. 过滤掉不健康的智能体
3. 多维评分：能力匹配度(+10)、上下文窗口匹配(+20/-不足扣分)、成本(-avg/10)、负载(-15)、优先级(+5)
4. 按评分降序排序，返回最佳匹配

**三个适配器实现**：

| 适配器 | 文件 | 接入方式 | 主要能力 |
|--------|------|----------|----------|
| `ClaudeCode适配器` | `适配器/claude_code.py` | Anthropic Async API | 代码生成、需求工程、架构设计、代码审查 |
| `OpenCode适配器` | `适配器/opencode.py` | CLI 子进程 | 代码生成、代码审查、代码探索 |
| `Codex适配器` | `适配器/codex.py` | OpenAI Async API | 代码生成、测试生成、代码解释 |

**安全创建适配器** (`基础.py:157`) 是关键的容错函数，捕获所有创建异常并返回 `None`，确保一个智能体创建失败不会影响其他智能体。

### 3.3 钩子

**路径**: `src/hermes/钩子/`

**职责**: 提供五层生命周期钩子系统，在不修改核心逻辑的情况下扩展系统行为。

**五层架构**:

```
┌─────────────────────────────────────────────────┐
│ 第1层：会话层 (Session)                          │
│ 生命周期事件、恢复、通知                          │
│ 7 个钩子：流水线启动/完成/失败通知、阶段开始/完成  │
│ 通知、任务开始/完成通知                           │
├─────────────────────────────────────────────────┤
│ 第2层：工具守护层 (Tool Guard)                   │
│ 工具执行前后守护                                  │
│ 4 个钩子：任务输入验证、权限检查、结果记录、错误   │
│ 处理                                             │
├─────────────────────────────────────────────────┤
│ 第3层：转换层 (Transform)                        │
│ 消息/上下文变换                                   │
│ 3 个钩子：阶段结果转换、上下文数据合并、任务输出   │
│ 格式化                                           │
├─────────────────────────────────────────────────┤
│ 第4层：继续层 (Continuation)                     │
│ 循环管理，重试                                    │
│ 3 个钩子：任务失败重试、无限循环检测、阶段失败恢复 │
├─────────────────────────────────────────────────┤
│ 第5层：技能层 (Skill)                            │
│ 技能生命周期                                      │
│ 2 个钩子：技能需求检测、技能执行评估               │
└─────────────────────────────────────────────────┘
```

**钩子组合器** (`组合器.py:14`) 负责组装五层钩子：
- `组合(上下文)`: 调用每层工厂的 `创建钩子()` 方法，组装所有钩子
- `触发钩子(钩子字典, 事件, *参数)`: 按优先级排序后触发匹配事件的所有钩子

**钩子基础定义** (`基础.py`) 包含：
- `钩子层级` 枚举：5 个层级
- `钩子事件` 枚举：15 种事件（流水线3、阶段3、任务4、审批3、智能体4、错误2 — 实际定义见代码）
- `钩子` 数据类：名称、层级、事件、回调、优先级
- `安全创建钩子()`: 安全创建钩子，失败返回 `None`

每层工厂都通过 `安全创建钩子()` 创建钩子实例，创建失败的钩子被过滤掉，不影响其他钩子。

### 3.4 配置

**路径**: `src/hermes/配置/`

**职责**: 多层级配置管理，支持 closest-wins 策略和安全深度合并。

**核心组件**:

| 组件 | 文件 | 职责 |
|------|------|------|
| `配置加载器` | `配置加载器.py:13` | 多层级配置加载和合并 |
| `应用配置` | `模式.py:66` | Pydantic v2 配置模式 |
| `安全深度合并` | `合并.py:10` | 防原型污染的深度合并 |

**配置优先级** (从低到高，后者覆盖前者)：

```
1. 默认值（代码内 Pydantic 默认值）
2. 系统级配置 (/etc/hermes/配置.jsonc)
3. 用户级配置 (~/.hermes/配置.jsonc)
4. 父目录配置 (<父目录>/.hermes/配置.jsonc)
5. 项目级配置 (<项目>/.hermes/配置.jsonc)
6. 环境变量覆盖 (HERMES_ 前缀)
```

**closest-wins 策略** (`配置加载器.py:104`)：从项目目录向上遍历到 HOME 目录，收集所有配置文件，反转列表使最接近项目的配置最后合并，从而具有最高优先级。

**安全深度合并** (`合并.py:10`) 的安全特性：
- 阻止 `__proto__`、`constructor`、`prototype` 键（防原型污染）
- 深度限制（默认 50 层）防止循环引用
- 列表使用集合并集策略（去重）
- 嵌套字典递归合并

**环境变量覆盖** (`配置加载器.py:177`) 支持：
- `HERMES_环境=development`
- `HERMES_数据库URL=postgresql://...`
- `HERMES_智能体_claude_code_模型=claude-3-opus`（嵌套配置覆盖）

**配置模式** (`模式.py:66`) 使用 Pydantic v2 定义完整的配置结构，包括智能体配置、类别配置、流水线默认值、后台任务配置、禁用列表等。

### 3.5 后台

**路径**: `src/hermes/后台/`

**职责**: 异步任务执行、并发控制、熔断器保护。

**核心组件**:

| 组件 | 文件 | 职责 |
|------|------|------|
| `后台任务管理器` | `管理器.py:12` | 异步任务执行、超时、状态跟踪 |
| `并发管理器` | `并发.py:8` | 按智能体控制最大并发数 |
| `熔断器` | `熔断器.py:16` | 故障自动检测和恢复 |

**后台任务管理器** (`管理器.py:12`) 是任务执行的入口：
1. `启动任务()`: 检查熔断器 → 检查并发限制 → 创建 asyncio 任务
2. `_执行任务()`: 带超时执行 → 成功记录到熔断器 → 释放并发许可
3. `等待任务完成()`: 轮询等待（100ms 间隔）

**并发管理器** (`并发.py:8`) 使用 `asyncio.Lock` 保护并发计数，按智能体 ID 隔离，每个智能体独立限制最大并发数（默认 5）。

**熔断器** (`熔断器.py:16`) 实现三态熔断：

```
     成功            失败 >= 阈值
关闭 ────→ 正常工作 ────→ 打开
                              │
                   恢复超时后  │
                              ↓
                          半开 ──→ 允许少量请求
                           │           │
                    失败    │   成功 >= 半开请求数
                           ↓           ↓
                         打开        关闭
```

| 参数 | 默认值 | 说明 |
|------|--------|------|
| 失败阈值 | 5 | 连续失败次数触发熔断 |
| 恢复超时 | 60s | 打开后等待恢复的时间 |
| 半开请求数 | 3 | 半开状态允许的测试请求数 |

### 3.6 模型

**职责**: SQLAlchemy 2.0 ORM 数据模型，映射到 PostgreSQL 数据库表。

**实体关系图**:


## 5. 设计模式

### 5.1 基于类别的智能体路由

**核心思想**：解耦"需要什么"和"用什么实现"。

用户通过 `category` 字段声明任务需求类别（如 `ultrabrain`、`deep`、`quick`），而非直接指定智能体。类别路由器维护一张路由表，映射每个类别到候选智能体优先级列表，并根据运行时状态（健康、负载、能力）动态选择最佳智能体。

**实现**：`类别路由器` (`智能体/类别.py:9`)

**优势**：
- 新增智能体只需注册到路由表，无需修改业务代码
- 智能体故障时自动降级到备选
- 负载均衡自然融入选择策略

### 5.2 工厂 + 依赖注入

**核心思想**：所有组件通过工厂函数创建，支持运行时依赖注入。

`智能体工厂` (`智能体/基础.py:143`) 是抽象工厂基类，每个智能体适配器都有对应的工厂实现。`智能体注册表` 支持注册工厂或直接注册实例，获取智能体时优先使用缓存实例，未命中则通过工厂延迟创建。

**优势**：
- 延迟初始化：智能体只在首次使用时创建
- 可测试性：测试时可直接注册 Mock 实例
- 生命周期管理：注册表统一管理实例

### 5.3 安全创建模式

**核心思想**：包装所有工厂创建，确保一个组件失败不会崩溃整个系统。

`安全创建适配器()` (`智能体/基础.py:157`) 和 `安全创建钩子()` (`钩子/基础.py:72`) 是该模式的两个实现。它们捕获所有异常、记录详细日志、返回 `None` 而非抛出异常。调用方过滤掉 `None` 值即可。

**应用场景**：
- 智能体创建：一个智能体 API Key 无效不影响其他智能体
- 钩子创建：一个钩子初始化失败不影响其他钩子
- 钩子执行：`钩子组合器.触发钩子()` 中单个钩子回调异常被捕获

### 5.4 多层级配置系统

**核心思想**：配置按从最不具体到最具体的层级叠加，closest-wins 策略使项目级配置覆盖用户级配置。

`配置加载器` (`配置/配置加载器.py:13`) 从 5 个层级收集配置，使用 `安全深度合并多层()` 合并。支持 JSONC 格式（带注释的 JSON），环境变量作为最高优先级覆盖。

**安全特性**：合并过程阻止原型污染键，深度限制防止循环引用。

### 5.5 五层钩子系统

**核心思想**：将系统生命周期钩子按关注点分层，每层独立创建、独立失败。

```
会话层     → "发生了什么"（通知、日志）
工具守护层 → "安不安全"（验证、权限、错误处理）
转换层     → "数据怎么变"（格式转换、上下文合并）
继续层     → "还要不要继续"（重试、循环检测、恢复）
技能层     → "需要什么技能"（技能检测、评估）
```

每层有独立的工厂（`会话钩子工厂`、`工具守护钩子工厂` 等），由 `钩子组合器` 统一组装。钩子按优先级排序执行，单个钩子异常不影响其他钩子。

### 5.6 后台任务管理器

**核心思想**：将任务执行与调用方解耦，集成并发控制和熔断器。

`后台任务管理器` (`后台/管理器.py:12`) 封装了三个关注点：
- **并发控制**：`并发管理器` 限制每个智能体的同时任务数
- **故障保护**：`熔断器` 在连续失败时自动断开
- **超时管理**：`asyncio.wait_for` 限制单个任务执行时间

三者组合形成完整的任务执行保障链：请求 → 熔断器检查 → 并发检查 → 执行 → 结果记录 → 释放资源。

## 6. 容错机制

### 6.1 安全创建

**覆盖范围**：智能体适配器创建、钩子创建

**机制**：`安全创建适配器()` 和 `安全创建钩子()` 捕获所有异常，返回 `None`，调用方过滤。

**效果**：一个组件初始化失败（如 API Key 无效、网络不可达）不会阻止系统启动，只是该组件不可用。

### 6.2 熔断器

**实现**：`熔断器` (`后台/熔断器.py:16`)

**三态模型**：
- **关闭**（正常）：所有请求通过，重置失败计数
- **打开**（熔断）：所有请求拒绝，等待恢复超时
- **半开**（试探）：允许少量请求，成功则关闭，失败则重新打开

**触发条件**：连续失败 5 次（可配置）

### 6.3 并发管理

**实现**：`并发管理器` (`后台/并发.py:8`)

**机制**：每个智能体维护独立的并发计数器，使用 `asyncio.Lock` 保护。超过最大并发数时拒绝新请求。

### 6.4 任务重试

**多层重试**：
1. **继续层钩子** (`钩子/继续/继续钩子.py:40`)：任务失败时检查重试次数，最多 3 次
2. **任务队列** (`队列/任务队列.py:147`)：队列中的任务失败后重新入队，最多重试 3 次
3. **状态机** (`编排器/状态机.py:20`)：`failed → running` 允许重新执行

### 6.5 循环检测

**两层防护**：
1. **DAG调度器** (`编排器/调度器.py:69`)：基于 Kahn 算法检测依赖循环，构建 DAG 时抛出 `依赖循环检测异常`
2. **继续层钩子** (`钩子/继续/继续钩子.py:64`)：运行时检测任务是否依赖于自身

### 6.6 超时控制

| 层级 | 默认值 | 说明 |
|------|--------|------|
| 任务超时 | 600s | 后台管理器中 `asyncio.wait_for` |
| 阶段超时 | 3600s | 配置模式中定义 |
| 流水线超时 | 86400s | 配置模式中定义 |
| 审批超时 | 300s | 人工界面服务，超时自动标记 |
| 健康检查 | 5s | OpenCode CLI 子进程超时 |
| 队列取任务 | 1s | 工作器循环中 `wait_for` 超时 |

## 7. 安全架构

### 7.1 RBAC 权限模型

**实现**：`认证服务` (`认证/RBAC.py:89`)

4 个角色覆盖 17 个权限，权限粒度到 `资源:操作` 级别（如 `pipeline:run`、`approval:decide`）。权限检查基于角色-权限映射表，一个用户可拥有多个角色，权限取并集。

### 7.2 API 密钥管理

**实现**：`认证服务.创建API密钥()` (`认证/RBAC.py:122`)

- 密钥格式：`hb_<secrets.token_urlsafe(32)>`
- 存储：SHA-256 哈希，原文不保存
- 有效期：默认 90 天，支持自定义
- 验证：哈希比对 + 激活检查 + 过期检查

### 7.3 安全配置合并

**实现**：`安全深度合并()` (`配置/合并.py:10`)

- 阻止 `__proto__`、`constructor`、`prototype` 键
- 深度限制 50 层，防止循环引用攻击
- `检查原型污染()` 函数可递归扫描配置字典

### 7.4 输入验证

**多层验证**：
1. **Pydantic 模式验证**：所有配置通过 Pydantic v2 模式验证，包括类型、范围约束
2. **工具守护层钩子** (`钩子/工具守护/工具守护钩子.py:45`)：验证任务定义包含必要字段（`id`、`name`、`type`）
3. **FastAPI 请求验证**：Pydantic 模型自动验证 API 请求体

### 7.5 速率限制

按角色限制 API 请求频率：

| 角色 | 限制 |
|------|------|
| 管理员 | 1000 req/min |
| 项目所有者 | 500 req/min |
| 开发者 | 200 req/min |
| 查看者 | 100 req/min |
